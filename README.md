## 1. Опыт тестирования

Ручное тестирование: более 2 лет коммерческого опыта в 3COMMAS. Проходил в компании стажировку и далее был устроен в штат. За это время изучил ряд инструментов, указанных в резюме. Дополнительно пользовался Graylog и Wireshark. Также в работе удалось немного поработать с Playwright MCP. В рамках тестовго задания работал с Postman по документации DummyJSON, анализ фактических ответов API и уточнение ожидаемого поведения.

Нагрузочное тестирование: базовый опыт есть, но в рамках этого задания нагрузочные проверки не выполнялись, потому что задача ограничена функциональными API-проверками через Postman. Для нагрузки отдельно можно было бы использовать JMeter, но это не является целью текущей коллекции.

Тестирование безопасности: базовый опыт есть на уровне API-проверок авторизации, токенов, отсутствия чувствительных данных в ответах и негативных сценариев без токена. В этой коллекции добавлена проверка, что `/auth/me` не должен возвращать пароль, и проверка отказа доступа без bearer token. Больше работал с Kafka UI.

## 2. Запуск

Установить зависимости:

```bash
npm install
```

Запустить тесты:

```bash
npm test
```

Команда из `package.json` запускает коллекцию через Newman:

```bash
newman run "API Tests - Auth and Carts.postman_collection.json" \
  --env-var baseUrl=https://dummyjson.com \
  --env-var username=emilys \
  --env-var password=emilyspass \
  --env-var invalidPassword=wrong_password \
  --env-var notFoundCartId=999999 \
  --reporters cli
```

В Postman коллекцию можно запускать вручную, но перед запуском нужно задать переменные:

| Переменная | Значение |
|---|---|
| `baseUrl` | `https://dummyjson.com` |
| `username` | `emilys` |
| `password` | `emilyspass` |
| `invalidPassword` | `wrong_password` |
| `notFoundCartId` | `999999` |

`accessToken`, `refreshToken`, `userId` и `cartId` заполняются автоматически.

## 3. Краткий список реализованных проверок

### Auth

1. `POST /auth/login` — успешная авторизация.
   Проверяется статус `200`, наличие `id`, `username`, `email`, `accessToken`, `refreshToken`. После успешного ответа токены и `userId` сохраняются в переменные окружения.

2. `POST /auth/login` — неверный пароль.
   Проверяется, что запрос отклоняется статусом `400` или `401`, а в ответе не возвращаются `accessToken` и `refreshToken`.

3. `POST /auth/login` — пустой username.
   Проверяется, что невалидные входные данные отклоняются, а токены не возвращаются.

4. `GET /auth/me` — запрос с bearer token.
   Проверяется статус `200`, наличие базовых полей пользователя, совпадение `id` с пользователем из авторизации и отсутствие поля `password` в ответе.

5. `GET /auth/me` — запрос без bearer token.
   Проверяется, что неаутентифицированный запрос отклоняется статусом `401` или `403`, а ответ содержит сообщение об ошибке. Перед запросом дополнительно очищается `Authorization` header и cookie jar, потому что DummyJSON может устанавливать токены в cookies.

### Carts

6. `GET /carts/user/{userId}` — получение корзин пользователя.
   Проверяется статус `200`, наличие массива `carts`, метаданных `total`, `skip`, `limit`, а также принадлежность каждой корзины запрошенному `userId`. Первый найденный `cartId` сохраняется для следующих запросов.

7. `GET /carts/{cartId}` — получение корзины по id.
   Проверяется статус `200`, совпадение `id`, наличие массива `products`, итоговых сумм `total`, `discountedTotal`, `totalProducts`, `totalQuantity`.

8. `POST /carts/add` — создание корзины.
   Проверяется статус `201`, наличие `id`, корректный `userId`, количество товаров, `totalProducts`, `totalQuantity`, `total` и `discountedTotal`.

9. `PATCH /carts/{cartId}` — обновление корзины.
   Проверяется статус `200`, совпадение `cartId`, наличие обновлённого продукта и ожидаемое количество товара.

10. `DELETE /carts/{cartId}` — удаление корзины.
    Проверяется статус `200`, флаг `isDeleted: true` и наличие `deletedOn`.

### Negative Carts

11. `POST /carts/add` — товар с `quantity: 0`.
    Ожидаемое поведение: API должен отклонить нулевое количество статусом `400` или `422`.

12. `POST /carts/add` — товар с отрицательным `quantity: -1`.
    Ожидаемое поведение: API должен отклонить отрицательное количество статусом `400` или `422`. Дополнительно проверяется консистентность суммы корзины.

13. `GET /carts/{notFoundCartId}` — несуществующая корзина.
    Проверяется статус `404` и наличие понятного сообщения об ошибке.

14. `POST /carts/add` — некорректный тип `userId`.
    Ожидаемое поведение: API должен отклонить строковый `userId` статусом `400` или `422` и не возвращать `500`.

## 4. Ограничения API

Уточню, что часть проверок специально оформлена как негативные сценарии, чтобы показать расхождение между ожидаемой бизнес-валидацией и поведением публичного мока API

### `/auth/me` возвращает чувствительные данные

В коллекции есть проверка, что ответ `/auth/me` не должен содержать поле `password`. Тест падает и это стоит оформить как потенциальный дефект в безопасности: клиентский API не должен возвращать пароль поле в ответе текущего пользователя. Даже если значение тестовое, в реальном продукте такое поведение недопустимо.

Ожидаемое поведение: `/auth/me` возвращает идентификатор, username, email и публичные данные профиля, но не возвращает пароль.

Фактическое поведение: API может вернуть поле `password`, из-за чего тест падает.

Что я бы уточнил у аналитика или backend-разработчика, если бы работал у вас: 
Является ли поле `password` частью mock dataset только для демонстрации или это считается контрактом API? Если это реальный контракт, поле нужно удалить из клиентского ответа или заменить на безопасный признак, например `hasPassword: true`.

### Нулевое или отрицательное количество товара может не отклоняться

Проверки `quantity: 0` и `quantity: -1` ожидают `400` или `422`, потому что с точки зрения бизнес-логики корзины количество товара должно быть положительным целым числом.

Ожидаемое поведение: API валидирует `quantity` и возвращает ошибку валидации.

Фактическое поведение: DummyJSON может принять такое тело и вернуть успешный ответ, потому что endpoint работает как мок-сервер и не всегда выполняет строгую бизнес-валидацию.

Что уточнить у аналитика или разработчика: 
Какие ограничения должны быть у `quantity`: минимальное значение, максимальное значение, допустимы ли дробные числа?
Как должна выглядеть ошибка валидации и какой статус должен возвращаться — `400` или `422`?

### Некорректный тип `userId` может не отклоняться

Проверка со строковым `userId: "abc"` ожидает ошибку `400` или `422`, потому что `userId` должен быть числовым идентификатором пользователя.

Ожидаемое поведение: API отклоняет некорректный тип поля и возвращает понятную ошибку.

Фактическое поведение: мокк API может не валидировать тип строго.

Что уточнить: должен ли бэкэнд выполнять строгуую типизацию полей по типам и какой формат ошибки считается стандартным?

### Create/update/delete не изменяют реальные данные на сервере

API симулирует изменение данных и возвращает рассчитанный ответ, но не сохраняет изменения на сервере. Поэтому после создания корзины нельзя ожидать, что она реально появится в общем списке, а после удаления нельзя проверять повторным GET, что корзина физически исчезла.

Такое ограничение связано с публичным характерном мокк API. В реальном проекте - это считалось бы багом.

## 5. Остальные проверки

Исходя из ТЗ задания, основной акцент сделан на 1–2 осмысленных проверки на метод: статус ответа, наличие ключевых бизнес-полей, сохранение и переиспользование токена, связь `userId` между авторизацией и `/auth/me`, связь `userId` и `cartId` между запросами корзины, отсутствие чувствительных данных, негативные сценарии на авторизацию, отсутствие токена, несуществующую корзину и невалидные данные. Это базовые проверки, которые были выполнены в рамках 1 часа работы.
